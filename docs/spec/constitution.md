# StoryForgeAI 项目宪法

**版本**: 1.0  
**生效日期**: 2026-04-02  
**适用范围**: 所有项目成员和 AI 助手

---

## 前言

本宪法定义了 StoryForgeAI 项目的核心原则、约束和决策规则，确保项目开发的一致性、质量和可持续性。所有开发决策必须遵循本宪法。

---

## 第一章：项目定位

### 第 1 条：项目性质

StoryForgeAI 是一个**学习 + 娱乐**项目，旨在探索 AI Agent 技术在小说创作领域的应用。

### 第 2 条：目标用户

- **主要用户**: 项目开发者本人（15 年 Java 经验）
- **潜在用户**: 对 AI 辅助创作感兴趣的开发者

### 第 3 条：核心价值

1. **技术学习**: 深入理解 AI Agent、RAG、MCP 等前沿技术
2. **实用价值**: 构建真正有用的小说创作辅助工具
3. **成本可控**: 每日预算 < 1 美元，避免过度消费

---

## 第二章：技术原则

### 第 4 条：技术选型原则

1. **Java First**: 核心代码必须使用 Java 21+
2. **成熟优先**: 优先选择成熟、社区活跃的框架
3. **渐进演进**: 初期简化实现，后期按需优化
4. **成本敏感**: 模型调用必须考虑成本

### 第 5 条：架构原则

1. **分层架构**: 用户接口层、API 网关层、Agent 编排层、工具层、记忆层、模型层
2. **职责分离**: 每个组件职责单一，避免上帝类
3. **接口抽象**: 核心接口自行抽象，降低框架耦合
4. **依赖倒置**: 高层模块不依赖低层模块，两者都依赖抽象

### 第 6 条：代码质量原则

1. **可读性优先**: 代码应自文档化，避免过度技巧化
2. **测试覆盖**: 核心逻辑必须有单元测试
3. **日志规范**: 所有模型调用、工具执行必须有 traceId
4. **异常处理**: 不吞没异常，必须有明确的错误处理策略

---

## 第三章：开发约束

### 第 7 条：技术栈约束

| 组件 | 强制要求 | 禁止使用 |
|------|---------|---------|
| 语言 | Java 21+ | Kotlin, Scala, Groovy |
| Agent 框架 | LangChain4j | 自研 Agent 框架（初期） |
| 向量数据库 | Milvus / Chroma | Elasticsearch（向量检索） |
| 模型 | GLM-4.5-Flash | GPT-4（成本过高） |

### 第 8 条：成本约束

1. **每日预算**: ≤ 1.0 USD
2. **模型选择**: 优先使用 GLM-4-Flash（低成本）
3. **Token 优化**: Prompt 应精简，避免冗余
4. **缓存策略**: 相同请求应缓存结果

### 第 9 条：性能约束

1. **首 token 延迟**: < 3 秒（P95）
2. **总响应时间**: < 30 秒（P95）
3. **工具执行时间**: < 5 秒（单个工具）
4. **向量检索延迟**: < 500ms

---

## 第四章：开发流程

### 第 10 条：迭代开发

项目采用**迭代式开发**，每个迭代 2-3 周：

- **迭代 1**: MVP（单模型 + 无状态 + 1 个工具）
- **迭代 2**: 记忆系统（三层记忆 + 3 个工具）
- **迭代 3**: MCP + 多模型（工具 MCP 化 + fallback）
- **迭代 4**: 优化与扩展（性能优化 + 新工具）

### 第 11 条：版本管理

1. **分支策略**: Git Flow（main, develop, feature/*）
2. **提交规范**: Conventional Commits（feat, fix, docs, refactor）
3. **代码审查**: 核心代码变更需要自我审查
4. **文档同步**: 代码变更必须同步更新文档

### 第 12 条：测试策略

1. **单元测试**: 核心业务逻辑必须有单元测试
2. **集成测试**: 关键流程（意图识别、工具调用）需要集成测试
3. **性能测试**: 每个迭代结束进行性能测试
4. **验收测试**: 按照验收标准进行功能验收

---

## 第五章：质量标准

### 第 13 条：代码质量标准

1. **命名规范**: 遵循 Java 命名规范，见名知意
2. **注释规范**: 复杂逻辑必须有注释，避免无意义注释
3. **代码格式**: 使用统一的代码格式化规则
4. **静态检查**: 使用 SpotBugs / SonarQube 进行静态检查

### 第 14 条：文档质量标准

1. **API 文档**: 使用 OpenAPI 规范
2. **架构文档**: 使用 Mermaid 图表
3. **用户文档**: README 必须包含快速开始指南
4. **变更日志**: CHANGELOG 记录所有重要变更

### 第 15 条：可观测性标准

1. **日志规范**: 
   ```
   [traceId={traceId}] {level} {component} - {message}
   ```
2. **指标收集**: 模型调用次数、Token 消耗、响应时间
3. **错误追踪**: 所有异常必须记录堆栈信息
4. **成本追踪**: 每次模型调用记录预估费用

---

## 第六章：决策规则

### 第 16 条：技术决策流程

1. **问题识别**: 明确需要决策的问题
2. **方案列举**: 列举至少 2 个可行方案
3. **利弊分析**: 分析每个方案的优缺点
4. **决策记录**: 记录决策理由和风险
5. **回顾优化**: 定期回顾决策，必要时调整

### 第 17 条：冲突解决

1. **技术冲突**: 以**简单性**和**可维护性**为准绳
2. **性能冲突**: 以**用户体验**为准绳
3. **成本冲突**: 以**预算限制**为准绳

### 第 18 条：变更管理

1. **小变更**: 直接实施，记录在 CHANGELOG
2. **中等变更**: 评估影响，更新文档后实施
3. **大变更**: 需要重新评估架构，更新技术方案

---

## 第七章：风险管理

### 第 19 条：技术风险

| 风险 | 缓解措施 |
|------|---------|
| LangChain4j API 变化 | 核心接口自行抽象 |
| Milvus 运维复杂 | Docker Compose + 监控 |
| GLM-4-Flash 质量不稳定 | 多模型 fallback |
| 成本超支 | 预算限制 + 告警 |

### 第 20 条：业务风险

| 风险 | 缓解措施 |
|------|---------|
| 用户需求变化 | 迭代式开发，快速响应 |
| 工具效果不佳 | 充分测试，持续优化 |
| 性能不达标 | 性能测试，优化瓶颈 |

---

## 第八章：禁止事项

### 第 21 条：禁止行为

1. **禁止硬编码**: 配置项必须外部化
2. **禁止吞没异常**: 所有异常必须有明确的处理策略
3. **禁止过度设计**: 遵循 YAGNI 原则
4. **禁止忽略成本**: 所有模型调用必须考虑成本
5. **禁止跳过测试**: 核心逻辑必须有测试覆盖

### 第 22 条：技术债务

1. **记录技术债**: 所有技术债务必须记录在 TODO
2. **定期清理**: 每个迭代预留时间清理技术债
3. **避免累积**: 技术债务不应超过 2 周
