# 需求澄清与概要设计

**项目**: 小说创作大师 Agent  
**版本**: 1.0  
**日期**: 2026-04-02

---

## 1. 需求澄清

### 1.1 核心问题定义

#### 问题陈述

小说作者在创作长篇小说时面临以下痛点：

1. **一致性维护困难**
   - 角色在不同章节的行为、性格可能出现矛盾
   - 时间线、地理位置容易出现逻辑漏洞（如角色瞬移）
   - 伏笔埋设后忘记回收，或回收时缺乏铺垫

2. **创作效率低下**
   - 构思角色弧光、情节转折需要大量时间
   - 分析已有内容的逻辑性需要反复阅读
   - 续写时需要回顾大量前文，容易遗漏细节

3. **缺乏智能辅助**
   - 现有工具多为文本编辑器，缺乏深度理解能力
   - AI 生成工具缺乏对整部小说的全局认知
   - 无法提供基于证据的分析建议

#### 解决方案

构建一个具备**三层记忆系统**的 AI Agent，通过：
- **长期记忆**（RAG）：存储整部小说的知识库，支持语义检索
- **工作记忆**：维护当前小说状态（角色位置、伏笔、冲突）
- **短期记忆**：保持对话上下文连贯

结合**工具调用能力**（角色弧光分析、情节漏洞检测、风格迁移），提供：
- 智能分析（角色立体度、情节逻辑）
- 辅助创作（续写、构思、润色）
- 一致性保障（状态追踪、伏笔管理）

### 1.2 用户故事

#### US-1: 角色立体度分析

**作为** 小说作者  
**我想要** 分析某个角色是否扁平  
**以便于** 改进角色塑造，增强人物深度

**验收标准**:
- Agent 能识别角色名称
- 从长期记忆检索角色相关片段
- 调用 CharacterArcPlanner 工具分析弧光
- 输出带证据的分析报告（引用具体章节）

---

#### US-2: 情节漏洞检测

**作为** 小说作者  
**我想要** 检测情节中的逻辑漏洞  
**以便于** 修复矛盾，提升小说质量

**验收标准**:
- Agent 能检测角色瞬移、动机缺失、时间线冲突
- 输出漏洞类型、严重程度、具体位置
- 提供修复建议

---

#### US-3: 智能续写

**作为** 小说作者  
**我想要** 根据前文续写下一章  
**以便于** 提高创作效率

**验收标准**:
- Agent 能加载工作记忆（角色位置、伏笔、章节目标）
- 检索上一章结尾片段，保证连贯性
- 生成符合风格约束的内容
- 自动更新工作记忆（新角色位置、新伏笔）

---

#### US-4: 风格迁移

**作为** 小说作者  
**我想要** 将某段文字改写为特定风格  
**以便于** 统一小说文风

**验收标准**:
- 支持多种风格（古风、现代、科幻、悬疑、幽默）
- 支持约束（保留关键词、禁止词、句长范围）
- 输出改写结果 + 变更说明

### 1.3 非功能性需求澄清

#### 性能需求

| 场景 | 性能目标 | 理由 |
|------|---------|------|
| 首token延迟 | < 3秒 | 用户体验要求，避免等待焦虑 |
| 工具执行 | < 5秒 | 单个工具调用不应成为瓶颈 |
| 向量检索 | < 500ms | Milvus 本地部署，应达到毫秒级 |
| 总响应时间 | < 30秒 | 包含工具调用 + 模型生成 |

#### 成本需求

- **每日预算**: 1.0 USD
- **理由**: 学习 + 娱乐项目，成本敏感
- **策略**: 使用 GLM-4-Flash（低成本） + 成本追踪 + 预算限制

#### 可扩展性需求

- **工具扩展**: 支持 MCP 协议，可接入外部工具
- **模型扩展**: 预留 OpenAI/Claude 适配接口
- **存储扩展**: 向量库、关系库可替换（接口抽象）

## 2. 概要设计

### 2.1 系统架构

#### 2.1.1 架构风格

**分层架构 + Agent 编排模式**

```
┌─────────────────────────────────────────┐
│           用户接口层 (CLI/Web)            │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│         API 网关层 (REST + SSE)          │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│       Agent 编排层 (LangChain4j)         │
│  ┌──────────────────────────────────┐   │
│  │  Orchestrator                    │   │
│  │  ├─ Intent Router                │   │
│  │  ├─ Tool Executor                │   │
│  │  └─ Memory Manager               │   │
│  └──────────────────────────────────┘   │
└─────────────────────────────────────────┘
                    ↓
┌──────────────┬──────────────┬───────────┐
│   工具层      │   记忆层      │  模型层   │
│  (Skill+MCP) │ (三层记忆)    │ (Gateway) │
└──────────────┴──────────────┴───────────┘
```

#### 2.1.2 核心组件

**1. AgentOrchestrator（编排器）**
- 职责：协调整个请求处理流程
- 输入：用户请求
- 输出：流式响应
- 关键方法：`handle(UserRequest) -> Flux<TokenChunk>`

**2. IntentRouter（意图路由）**
- 职责：识别用户意图（BRAINSTORM/POLISH/ANALYZE/CONTINUE/GENERAL_CHAT）
- 实现：阶段一用规则匹配，阶段二用小模型分类

**3. ToolExecutor（工具执行器）**
- 职责：调用内部工具（Skill）或外部工具（MCP）
- 关键方法：`execute(ToolCall) -> ToolResult`

**4. MemoryManager（记忆管理器）**
- 职责：构建上下文、持久化交互记录
- 关键方法：`buildContext() -> ContextBundle`, `persistTurn()`

**5. ModelGateway（模型网关）**
- 职责：统一模型调用接口、fallback、成本追踪
- 关键方法：`chatStream() -> Flux<ModelChunk>`

### 2.2 数据流设计

#### 2.2.1 核心数据流

```
用户输入
  ↓
意图识别
  ↓
上下文构建 ← [短期记忆 + 长期记忆 + 工作记忆]
  ↓
工具调用（可选）
  ↓
模型生成
  ↓
流式输出
  ↓
状态持久化 → [短期记忆 + 长期记忆 + 工作记忆]
```

#### 2.2.2 数据流向

**请求流**:
```
UserRequest → IntentType → ContextBundle → ToolCalls → ModelRequest → ModelChunks
```

**响应流**:
```
ModelChunks → TokenChunks → SSE Stream → User
```

**状态流**:
```
TurnRecord → ShortTermMemory + LongTermMemory + WorkingState
```

### 2.3 关键设计决策

#### 决策 1: Agent 框架选择

**选项**:
- LangChain4j
- Spring AI
- 自研

**决策**: **LangChain4j**

**理由**:
1. Agent 抽象成熟（Tool/Memory/Chain）
2. Java 生态友好
3. 社区活跃，案例丰富
4. 与 Spring Boot 集成良好

**风险**:
- 版本迭代快，API 可能变化
- 文档相对 Python 版本较少

**缓解**:
- 锁定版本（0.35+）
- 核心接口自行抽象，降低耦合

---

#### 决策 2: 向量数据库选择

**选项**:
- Milvus（本地）
- Pinecone（云）
- Chroma（轻量）

**决策**: **Milvus 本地版**

**理由**:
1. 本地可控，无网络延迟
2. 成本低（无云服务费）
3. 支持大规模数据（未来扩展）
4. Docker 部署简单

**风险**:
- 运维复杂度（相比云服务）
- 需要学习 Milvus 特有语法

**缓解**:
- 使用 Docker Compose 简化部署
- 抽象 VectorStore 接口，便于替换

---

#### 决策 3: 记忆分块策略

**选项**:
- 按 token 数分块（固定 500 tokens）
- 按段落分块
- 按场景分块

**决策**: **按场景分块**

**理由**:
1. 场景是小说的语义单元，检索更精准
2. 包含完整的时间、地点、人物、事件信息
3. 便于后续分析（情节漏洞检测）

**风险**:
- 场景识别需要额外处理
- 场景长度不均，可能超出 embedding 限制

**缓解**:
- 使用规则 + 模型混合识别场景边界
- 超长场景拆分为多个 chunk，共享 metadata

---

#### 决策 4: 工具调用方式

**选项**:
- 全部进程内调用（Skill）
- 全部 MCP 调用（外部进程）
- 混合模式（核心工具进程内，扩展工具 MCP）

**决策**: **混合模式**

**理由**:
1. 核心工具（CharacterArcPlanner、PlotGapDetector）进程内调用，低延迟
2. 扩展工具（未来）通过 MCP 接入，灵活扩展
3. 渐进式演进：初期全部进程内，后期按需 MCP 化

**风险**:
- 需要维护两套调用逻辑

**缓解**:
- 统一 Tool 接口，屏蔽调用方式差异
- ToolExecutor 内部判断 viaMcp 标志

---

#### 决策 5: 角色驱动情节生成（创新点 1）

**选项**:
- 传统上帝视角生成（模型一次性生成整章）
- 角色驱动生成（每个角色独立决策，情节涌现）

**决策**: **角色驱动生成（渐进式引入）**

**理由**:
1. 角色行为更符合性格逻辑，情节转折更自然
2. 类似《模拟人生》的叙事涌现，创新性强
3. 适合复杂多角色叙事

**风险**:
- 实现复杂度高，需要角色状态管理
- 多角色冲突解决需要精心设计
- 成熟度较低（3/5），需要研究验证

**缓解**:
- 迭代 4+ 引入，先完成核心功能
- 从单角色模拟开始，逐步扩展到多角色
- 参考《模拟人生》和学术研究

---

#### 决策 6: 符号规划 vs 纯神经生成（创新点 3）

**选项**:
- 纯神经生成（LLM 直接生成文本）
- 符号规划 + 神经生成（PDDL 规划 + LLM 渲染）
- 混合模式（简单场景神经生成，复杂场景符号规划）

**决策**: **混合模式**

**理由**:
1. 符号规划保证空间/时间一致性，避免"角色瞬移"
2. 神经生成保证文本流畅性和创意性
3. 成熟度高（4/5），有成熟的 PDDL 库可用

**风险**:
- 需要学习 PDDL 语法
- 规划器可能生成不合理的行动序列

**缓解**:
- 使用 Java SymbolicPlanning 库
- 规划结果由 LLM 审查和调整
- 初期仅用于复杂场景（多角色、多地点）

---

#### 决策 7: 因果依赖图存储（创新点 4）

**选项**:
- 不存储依赖关系（用户修改时重新生成全部）
- 存储因果依赖图（支持最小回滚）

**决策**: **存储因果依赖图（迭代 5+）**

**理由**:
1. 大幅提升用户控制感，减少重复生成成本
2. 支持精细化编辑，类似 Git rebase
3. 创新性强（5 星），但成熟度低（2/5）

**风险**:
- 依赖关系识别准确率待验证
- 存储和查询复杂度高
- 实现难度大

**缓解**:
- 迭代 5+ 引入，先验证可行性
- 使用图数据库（Neo4j）存储依赖关系
- 从简单依赖（时间顺序）开始，逐步扩展

---

#### 决策 8: 约束求解器集成（创新点 5）

**选项**:
- 不使用约束求解器（模型自行折中）
- 使用约束求解器（OptaPlanner/Choco）

**决策**: **使用 OptaPlanner（迭代 3-4）**

**理由**:
1. 满足复杂约束（字数、对话次数、伏笔回收）
2. 自动处理约束冲突，输出可行解空间
3. 成熟度高（4/5），OptaPlanner 生态完善

**风险**:
- 需要学习 OptaPlanner 配置
- 约束建模可能复杂

**缓解**:
- 从简单约束开始（字数限制）
- 提供约束模板，降低使用门槛
- 参考官方文档和案例

---

#### 决策 9: 反事实推理引擎（创新点 2）

**选项**:
- 不使用反事实推理（依赖模型灵光一现）
- 使用反事实推理（系统化伏笔设计）

**决策**: **使用反事实推理（迭代 5+）**

**理由**:
1. 伏笔设计系统化，情节严密性大幅提升
2. 创新性强（4 星），但成熟度低（2/5）
3. 适合专业作家和复杂情节

**风险**:
- 需要轻量级推理模型（本地 Llama 3）
- 推理路径可能不准确
- 实现难度高

**缓解**:
- 迭代 5+ 引入，先验证可行性
- 使用本地 Llama 3 降低成本
- 从简单情节开始验证
